# 센터 대표 전용 CRM 포털 구축을 위한 단계별 프롬프트 (독립 포털 방식)

이 프롬프트 시리즈는 기존 '트레이너 앱' 소스코드에 기능을 덧붙이는 것이 아니라, **별도의 라우트(/gym-owner)와 전용 관리 시스템**을 구축하는 것을 목표로 합니다.

---

## 1단계: 독립 포털을 위한 DB 인프라 및 권한 설계
"현재 프로젝트에 센터 대표(Gym Owner)를 위한 전용 관리 인프라를 구축할 거야. 트레이너 앱과는 별개의 독립된 포털로 작동해야 해."

1. **Schema 확장**: `gyms` 테이블(센터 정보)을 신설해줘.
2. **관계 설정**: `trainers`와 `members` 테이블에 `gym_id` (FK)를 추가하고, 트레이너에게는 `employment_type` (정직원/프리랜서) 필드를 추가해줘.
3. **RBAC(다중 권한)**: `community_users` 테이블의 역할을 활용하거나 새로운 권한 테이블을 설계하여, 한 유저가 특정 센터의 '대표' 권한을 가질 수 있게 해줘.
4. **독립성 유지**: `gym_id`가 없는 트레이너나 회원은 기존처럼 개인 모드로 작동하고, `gym_id`가 연결된 경우에만 센터 CRM 포털에서 조회되도록 설계해줘.
5. **Prisma 마이그레이션**: 위 내용을 반영한 Prisma Schema를 작성해줘.

---

## 2단계: 독립된 CRM 포털 경로(/gym-owner) 및 뼈대 생성
"이제 DB 인프라를 바탕으로 대표님만 접속할 수 있는 전용 웹 포털 화면을 만들 거야."

1. **Route 생성**: `App.jsx`에 `/gym-owner` 경로를 추가하고, 전용 컴포넌트인 `GymOwnerPortal.jsx`를 생성해줘.
2. **UI 레이아웃**: 기존 트레이너 앱(Dark)이나 회원 포털(Light)과는 차별화된, 전문 경영 관리 도구 느낌의 깔끔한 대시보드 레이아웃을 잡아줘. (사이드바 메뉴: 대시보드, 직원관리, 회원관리, 매출/정산)
3. **접근 제어**: 유저의 역할이 `gym_owner`가 아니거나 해당 센터 권한이 없으면 접속을 차단하는 보호 로직을 넣어줘.
4. **스타일링**: `styles/gym-owner.css`를 별도로 생성하여 기존 스타일과 충돌하지 않게 해줘.

---

## 3단계: 프라이버시 보호를 포함한 데이터 연동 (Privacy Wall)
"대표 포털에서 소속 트레이너의 데이터를 볼 수 있게 하되, 민감한 개인 코칭 내용은 철저히 보호해야 해."

1. **Data Fetching**: 센터 ID(`gym_id`)를 기반으로 소속된 모든 트레이너와 회원의 정보를 불러오는 API 로직을 작성해줘.
2. **보안 가이드라인**: 대표는 수업의 '일시', '횟수', '상태'는 볼 수 있지만, `logs.content` 필드(수업의 세부 대화 내용 및 피드백)는 절대 열람할 수 없도록 데이터 필터링 계층을 만들어줘.
3. **승인 프로세스**: 트레이너가 앱 설정에서 '센터 데이터 연동 승인'을 한 경우에만 대표 CRM에 데이터가 노출되도록 `trainer_gym_link` 상태 관리 로직을 구현해줘.

---

## 4단계: 센터 경영 전용 관리 기능 구현 (HR 및 재무)
"마지막으로 전문 CRM 기능을 독립 포털에 구현해줘."

1. **직원 관리**: 트레이너별 수업 소진율, 재등록률, 이번 달 예상 정산금을 한눈에 보는 리스트 뷰를 만들어줘.
2. **통합 매출 엔진**: 센터 전체의 결제 내역(`payments`)을 합산하고, 트레이너 인센티브를 제외한 '센터 순수익'을 자동 계산하는 대시보드 위젯을 작성해줘.
3. **자동화 알림**: 센터 전체 회원 중 '만료 임박' 혹은 '장기 미출석' 회원을 자동으로 분류하여 대표에게 리스트업 해주는 기능을 추가해줘.
4. **단순화된 UX**: 모든 경영 지표는 복잡한 표 대신 요약 카드와 그래프로 표현하여 1인 대표가 운영하기 편리하도록 단순화해줘.
