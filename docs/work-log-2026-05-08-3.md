## 머지된 PR 이력

### PR #19 — admin RPC 복원 (`feat(admin)`) — `052_admin_rpcs.sql`

**배경:**
이전 세션에서 머지된 050/051 의 `auth.uid()` 기반 strict RLS 적용 후 AdminPortal 의 anon key 직접 access 가 침묵 실패하던 5개 흐름을 SECURITY DEFINER RPC 로 복원.

**문제 정리 (적용 전 상태):**
- `trainers / members / logs` 직접 SELECT → anon key + auth.uid() IS NULL 조합으로 0행 반환
- `trainers` UPDATE (CRM 권한 토글, lines 753/762) → UPDATE 정책 부여 안 됨 → 무음 실패
- `trainers` INSERT (사전 등록, line 820) → INSERT 정책 부여 안 됨 → 실패
- 기존 `app_settings_admin_upsert` 의 토큰 검증이 `<> ''` 라 빈 문자열만 통과하는 명백한 버그
- 클라이언트 `'own-admin-123'` 하드코딩 + `p_secret`/`p_admin_token` 시그니처 혼재
- `admin_add_credits` / `admin_set_trainer_plan` 은 SECURITY DEFINER 인데 토큰 검증 자체가 없어 anon 누구나 호출 가능 (보안 약점)

**해결:**
- 마이그레이션 `052_admin_rpcs.sql` 한 파일에 통합:
  1. `app_settings` RLS 흡수 (`fix_rls_top3.sql` 의 ① 섹션 idempotent 흡수)
  2. `_admin_assert(p_admin_token text)` 공통 가드 함수 — 본문 하드코딩 토큰 비교
  3. 신규 RPC 6개: `admin_list_trainers`, `admin_list_members`, `admin_list_logs`, `admin_list_payments`, `admin_update_trainer_crm_permissions`, `admin_register_trainer`
  4. 기존 RPC 4개 패치: `app_settings_admin_upsert/delete` (시그니처 통일 + 토큰 검증 정상화), `admin_add_credits` / `admin_set_trainer_plan` (토큰 검증 추가)
- 모든 admin RPC 가 단일 토큰 (`VITE_ADMIN_DB_TOKEN`) 으로 통일
- `apps/admin/src/pages/AdminPortal.jsx` 의 5개 깨진 흐름을 RPC 호출로 교체
- `docs/migrations/052_apply_guide.md` 신규 — placeholder 치환 → SQL Editor 적용 → .env 주입 → 검증 절차 + 토큰 회전 SOP

**적용 절차 (사용자 검증 완료):**
1. PowerShell 로 32자 랜덤 토큰 생성 (`qnYZi...`)
2. `052_admin_rpcs.sql` 의 `<<ADMIN_TOKEN>>` placeholder 를 토큰으로 치환한 사본 생성
3. Supabase SQL Editor 에 붙여넣기 → Run → 에러 없음 확인
4. main repo `.env` 의 `VITE_ADMIN_DB_TOKEN` 갱신 (옛 `!eoghkaptiruscales2684!` → 새 토큰)
5. 검증 쿼리 통과 확인:
   - `SELECT admin_list_trainers('wrong')` → `42501: admin: unauthorized` ✓
   - `SELECT * FROM admin_list_trainers('<토큰>') LIMIT 1` → 1행 반환 ✓
6. `npm run dev:admin` → 모든 탭 동작 검증

**위험 / 한계:**
- 토큰이 Vite 빌드 산출물에 그대로 박힘 → admin 화면 외부 공개 시 devtools 로 추출 가능. 단기 완화: admin 도메인 사내망/IP 제한. 중장기 (Phase E 이후): Edge Function + httpOnly cookie 세션.

### PR #20 — 랜딩 v1/legacy 키 충돌 fix (`fix(landing)`)

**배경:**
PR #19 머지 후 사용자 동작 검증 중, "admin 에서 랜딩 콘텐츠 수정해도 실제 랜딩 페이지에 반영 안 됨" 12개 섹션 (메인 7 + CRM 4 + 커뮤니티 1) 버그를 사용자가 보고. 추적 결과 PR #19 와 무관한 pre-existing 버그 2개로 root cause 판정.

**Root cause A — 메인 랜딩 7개 섹션 (`apps/portal/src/pages/Landing.jsx`):**
- useEffect 가 단계 (1) 에서 `landing_v1` 통합 객체를 적용한 뒤, 단계 (2) 의 레거시 파편 키 (`landing_solutions`, `landing_reviews`, `landing_faqs`, `landing_comparison`, `landing_member_features`, `landing_plans_landing`, `landing_ai_highlight`) 를 **무조건 setX 로 덮어씀**.
- 의도(주석 기준)는 "v1 미존재일 때만 폴백" 이었으나 실제론 항상 덮어쓰기 → admin SoT 가 즉시 stale 데이터로 회귀.
- 기존엔 `portalsApplied` 플래그가 portal_buttons 에만 단독 가드 → 다른 12개 섹션은 무방비.

**Root cause B — CRM 랜딩 4개 섹션 (`apps/crm/src/pages/GymPortal.jsx`):**
- `landing_v1` 자체를 **읽지 않음**. 레거시 `landing_crm_hero / landing_crm_features / landing_crm_painpoints / landing_crm_roadmap` 만 .in() 키 목록에 있음.
- AdminPortal SoT 인 `landing_v1.crm_*` 와 영구 단절 → admin 의 CRM 섹션 수정이 영원히 stale.

**해결:**
- `Landing.jsx`: `v1Applied` Set 으로 단계 (1) 에서 적용된 섹션을 추적, 단계 (2) 의 모든 레거시 분기에 `!v1Applied.has(...)` 가드 추가. 13개 섹션 (hero/stats/problems/solutions/reviews/kakao/targets/member_features/plans_landing/faqs/comparison/ai_highlight/portal_buttons) 모두 일관 패턴.
- `GymPortal.jsx`: `landing_v1` 을 .in() 키 목록에 추가, `parseValue` 안전 파싱 도입, v1.crm_* 우선 적용 + 레거시 폴백 가드 동일 패턴.

**별개 이슈 (PR #20 미포함):**
- **커뮤니티 랜딩 페이지 부재:** 사용자 보고로 발견. admin 에 "커뮤니티 랜딩" UI 가 있으나 `community_hero` 를 읽는 코드가 portal/crm/admin 어디에도 없음. 처리 방향 (admin UI 숨김 vs 페이지 신규 구현) 결정 필요 → 후속 이슈로 분리.

**검증 (사용자 완료):**
- 12개 섹션 모두 admin 수정 → 랜딩 반영 확인.
- `npm run build` 3개 앱 모두 통과.

## 추가 배경 — 디버깅 흐름 기록

본 세션의 표면적 작업은 PR #19 한 건이지만, 사용자 검증 단계에서 12개 추가 버그가 드러나는 흐름을 거쳐 PR #20 까지 묶어 처리. 향후 유사 패턴 (RLS 또는 컨버전스 후 동작 확인 단계) 진단의 참고 사례:

1. **PR #19 코드 작성 → 빌드 → 머지** (계획대로 30분)
2. **사용자 SQL 적용 + dev 검증 진행 중** 12개 admin/landing 버그 보고 (예상치 못한 분기)
3. **버그 분석:** "12개가 같은 패턴 (admin 저장 → 랜딩 미반영) → 단일 root cause 가능성" 가설 → 보고된 시점이 PR #19 머지 전후 인지 사용자 확인 → "머지 전부터" 답변으로 052 무관 확인
4. **Code path 추적:**
   - admin 저장: `saveLandingV1Bundle` → `adminUpsertSetting('landing_v1', bundle)` 단일 키 통합 저장
   - 메인 랜딩 read: `Landing.jsx` 가 v1 적용 후 레거시 무조건 덮어쓰기 → root cause A
   - CRM 랜딩 read: `GymPortal.jsx` 가 v1 미독 → root cause B
5. **PR #20 적용 → 빌드 → push → 머지 → 사용자 검증**

교훈: **migration PR 후엔 사용자 검증을 마치 별도 phase 처럼 일정에 포함**. 예상치 못한 인접 버그 발견 가능성이 높음 (특히 RLS / SoT 통합 같은 cross-cutting 변경 후).
